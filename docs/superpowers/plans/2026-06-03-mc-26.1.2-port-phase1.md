# MC 26.1.2 (NeoForge) Port — Phase 1: Server-Side Runtime Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate the entire server-side runtime (~66 of 72 files) from Forge 1.20.1 to NeoForge 26.1.2, in dependency order with a green build at every step, ending with an in-game-verified Steve that spawns, responds to `/steve`, runs a basic action, and reaches Claude.

**Architecture:** The legacy source is staged under `port-staging/`. We reintroduce it bottom-up: MC-free pure-Java packages first (move unchanged), then the MC/Forge-coupled files migrated to NeoForge APIs. Migration of MC-internal APIs is **compile-driven** — we compile against the real decompiled NeoForge 26.1.2 sources already on disk and resolve exact symbols from the compiler + local sources + NeoForge docs, rather than guessing package paths.

**Tech Stack:** NeoForge 26.1.2.71, Java 25, ModDevGradle 2.0.141. Verified framework mappings: `DeferredRegister`/`DeferredRegister.Entities` + `DeferredHolder` (replaces `RegistryObject`); `ModConfigSpec` + `ModContainer#registerConfig` (replaces `ForgeConfigSpec`/`ModLoadingContext`); `@EventBusSubscriber` + `NeoForge.EVENT_BUS` game bus + mod bus from constructor (replaces `@Mod.EventBusSubscriber`/`MinecraftForge.EVENT_BUS`).

**Build command (this machine):** `./gradlew build` — system JDK 25 is the default, no `JAVA_HOME` needed.

---

## Sub-stage order & verification gates

> **Revised after dependency analysis (commit-time):** Beyond Stage 1a, the remaining
> server-side packages form one mutually-dependent component — `entity ↔ action ↔
> action/actions ↔ execution ↔ memory ↔ llm(remainder) ↔ event(bus) ↔ plugin` all
> reference each other (e.g. `memory.WorldKnowledge`→`SteveEntity`, `execution`→`action`+
> `entity`, `llm.TaskPlanner`→`entity`+`memory`). There is **no independently-compilable
> bottom-up layering**. The only isolated remaining piece is `config/SteveConfig`. So 1b/1c/1d
> collapse into: migrate config standalone, then migrate the whole tangled core as one unit.

| Stage | Scope | Verify |
|-------|-------|--------|
| **1a** ✅ | MC-free LLM stack (`di`, `llm/async`, `llm/resilience`) + build deps + tests | green; async tests pass |
| **1b** | `config/SteveConfig` standalone: `ForgeConfigSpec`→`ModConfigSpec` | compiles green |
| **1c** | The tangled server-side core (all remaining server packages) + full `SteveMod` wiring, migrated to compile green as one unit; per-package commits | `./gradlew build` green; **in-game**: spawn + `/steve` + basic action + Claude |

Stage 1c has no intermediate independent green-build gate (the component only compiles once whole); progress is tracked by shrinking the compiler error set, with per-package commits. Task-level detail for 1c is authored once 1b is green (the compiler then drives the real API deltas). This document fully details **1a** (done) and **1b**, and gives the **1c roadmap** (files, known mappings, methodology, gate).

## STAGE 1b — config/SteveConfig standalone (detailed)

`config/SteveConfig` has no intra-mod dependencies, so it migrates and compiles on its own. It exercises the NeoForge config API ahead of the big Stage 1c. Migration is a class-name swap: `net.minecraftforge.common.ForgeConfigSpec*` → `net.neoforged.neoforge.common.ModConfigSpec*`. The `SPEC` field type, `Builder`, `ConfigValue<T>`, `IntValue`, `DoubleValue`, `BooleanValue` are all nested under `ModConfigSpec`. The builder API (`comment`/`push`/`pop`/`define`/`defineInRange`) is unchanged. Registration stays in `SteveMod` (Stage 1c), not here.

**Files:** Move `port-staging/java/com/steve/ai/config` → `src/main/java/com/steve/ai/config`.

- [ ] **Step 1: Move the package back**

```bash
git mv port-staging/java/com/steve/ai/config src/main/java/com/steve/ai/config
```

- [ ] **Step 2: Migrate imports/types in `SteveConfig.java`**

Replace `import net.minecraftforge.common.ForgeConfigSpec;` with `import net.neoforged.neoforge.common.ModConfigSpec;`, and replace every `ForgeConfigSpec` token with `ModConfigSpec` (the `SPEC` field, the `Builder`, and the `ConfigValue`/`IntValue`/`DoubleValue`/`BooleanValue` types). Keep the `[ai]`/`[anthropic]`/`[openai]`/`[behavior]` sections and all default values intact (Claude default, model `claude-haiku-4-5`, etc.).

## STAGE 1c — Tangled server-side core + SteveMod wiring (roadmap)

This is the heart of the port: the one mutually-dependent component that only compiles as a whole. Authored into concrete tasks once 1b is green and the compiler reveals the real deltas.

**Files (all remaining server-side):** `event/` (bus + data + `ServerEventHandler`), `di/`-consumers, `plugin/` (5), `execution/` (10), `memory/` (3), `action/` (4), `action/actions/` (10), `llm/` remainder (`PromptBuilder`, `ResponseParser`, `TaskPlanner`, `OpenAIClient`, `GeminiClient`, `GroqClient`), `util/` (1), `structure/` (3), `entity/` (2), `command/` (1), and full `SteveMod` wiring. Plus the matching tests (`memory/WorldKnowledgeTest`, `structure/StructureGeneratorsTest`, `llm/TaskPlannerTest`, `action/ActionExecutorTest`).

**Methodology:** `git mv` the whole remaining server-side back at once, then drive `./gradlew compileJava` iteratively — fix `net.minecraftforge.*`→`net.neoforged.*` and changed `net.minecraft.*` calls against the compiler errors + local decompiled sources, shrinking the error set until green. Commit per package as it stops producing errors. No independent intermediate green gate (the component compiles only when whole).

**Verified framework migrations:**
- **SteveMod entry:** entity registration via `DeferredRegister.create(BuiltInRegistries.ENTITY_TYPE, MODID)` (or `DeferredRegister.Entities`); `RegistryObject` → `DeferredHolder`/`Supplier`. Register the DR to the **mod bus** (constructor param `IEventBus`, replacing `FMLJavaModLoadingContext.get().getModEventBus()`). Register config via injected `ModContainer#registerConfig(ModConfig.Type.COMMON, SteveConfig.SPEC)`. Attribute (`EntityAttributeCreationEvent`) and command (`RegisterCommandsEvent`) listeners on the appropriate bus. Drop the client-only `SteveGUI` registration (moves to Phase 2).
- **ServerEventHandler:** `@Mod.EventBusSubscriber` → `@EventBusSubscriber(modid = SteveMod.MODID)`; `@SubscribeEvent` from `net.neoforged.bus.api.SubscribeEvent`; `PlayerEvent.PlayerLoggedInEvent/PlayerLoggedOutEvent` from the NeoForge event package; register on `NeoForge.EVENT_BUS` (game bus).
- **SteveEntity:** `PathfinderMob`, `createMobAttributes()`/`AttributeSupplier`, AI `Goal` classes; `SynchedEntityData`/`EntityDataAccessor` — `defineSynchedData` now takes a `SynchedEntityData.Builder` (changed in 1.20.5); NBT `addAdditionalSaveData`/`readAdditionalSaveData`. Verify each against local sources.
- **Actions / world interaction:** block placement / `BlockState` / `Level#setBlock`, inventory & `ItemStack`/**data components** (NBT item data replaced 1.20.5+), crafting/recipe access, navigation, mob targeting — the most likely 26.1 deltas.
- **SteveCommands:** Brigadier + `Commands.literal/argument` — largely stable; fix `CommandSourceStack`/registration signature deltas.
- **Packaging note (later):** non-MC libs (resilience4j, caffeine, commons-codec, graalvm) work on the dev `runClient` classpath but need Jar-in-Jar for a production artifact — out of scope for the in-game gate, tracked for a packaging pass.

**Gate (Phase 1 exit):**
- `./gradlew build` green.
- User runs `./gradlew runClient`; in-game: a Steve spawns (on login / via command), `/steve <command>` is accepted, a basic action executes, and the log shows a Claude request/response (configured `[anthropic].apiKey`) — or a clean auth error if unconfigured.
