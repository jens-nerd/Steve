# Design: Port Steve auf Minecraft 26.1.2 (NeoForge)

**Datum:** 2026-06-03
**Status:** Approved

## Ziel

Steve von **Forge 1.20.1 / Java 17** auf **Minecraft 26.1.2** mit dem **NeoForge**-Loader
(`26.1.2.71`) lauffähig machen. 26.1.2 ist die Fortführung der 1.21.x-Reihe (Vorgänger war
1.21.11) unter dem neuen `year.drop.hotfix`-Versionsschema und erfordert **Java 25**.

## Rahmen / Entscheidungen

- **Loader:** NeoForge `26.1.2.71` (Ökosystem-Standard seit 1.20.5+, beste aktuelle Doku).
  Bedeutet Wechsel von `net.minecraftforge.*` → `net.neoforged.*`, eigener Config-/
  Event-Bus-API.
- **Strategie:** **Phasenweise** Migration mit In-game-Verifikation am Ende jeder Phase.
- **Build:** **ModDevGradle (MDG)**, geseedet aus dem offiziellen Template
  `NeoForgeMDKs/MDK-26.1.2-ModDevGradle`, ersetzt die bisherige ForgeGradle-Konfiguration.
- **Wissensgrenze:** Modell-Cutoff Jan 2026; 26.1-spezifische APIs werden gegen die Live-
  NeoForge-Doku (docs.neoforged.net, context7) + MDK verifiziert. Mehrere Compile-Fix-
  Iterationen pro Phase sind erwartbar.

## Gewählter Ansatz

Build auf das offizielle 26.1.2-MDK (ModDevGradle) re-basen, danach den bestehenden
`src`-Baum phasenweise migrieren. Vorteil: korrekte Toolchain/Wrapper garantiert → wir
kämpfen gegen Code-Probleme, nicht gegen Build-Konfig.

Verworfene Alternativen:
- **Hand-Konvertierung der ForgeGradle-Build** — höheres Risiko falscher Versionen, langsamer
  zum grünen Build.
- **Kompatibilitäts-Shim-Layer** — versionsübergreifende MC-API-Änderungen (Data Components,
  Rendering) lassen sich nicht sinnvoll shimmen; zusätzliche Abstraktion gegen das
  Simplicity-Prinzip.

## Phasenplan

Jede Phase endet mit einer In-game-Verifikation durch den Nutzer (MC 26.1.2/Java 25 kann
in dieser Umgebung nicht gestartet werden).

### Phase 0 — Fundament / Machbarkeits-Gate

- ForgeGradle → ModDevGradle (aus offiziellem 26.1.2-MDK); Java-Toolchain 17 → 25;
  Gradle-Wrapper-Bump.
- `META-INF/mods.toml` → `META-INF/neoforge.mods.toml` (NeoForge-Dependency, loaderVersion,
  MC-Range `[26.1.2,…)`). Den aktuell **doppelt einkopierten** `mods.toml`-Inhalt bereinigen
  (Datei enthält zwei aneinandergehängte Kopien).
- `pack.mcmeta` `pack_format` anheben.
- **GraalVM-`polyglot:23.1.0` prüfen** (genutzt von `CodeExecutionEngine`): Java-25-
  kompatible Version verfügbar? Dies ist die einzige Dependency, die einen Versions-Bump
  oder eine Neubewertung erzwingen könnte.
- **Verify:** `./gradlew build` erfolgreich **und** NeoForge-26.1.2-Client startet mit der
  (noch funktionslosen) Mod geladen.

### Phase 1 — Kern-Runtime

- `SteveMod` (Entry: `@Mod`-Konstruktor, NeoForge-`IEventBus`-Busse, `DeferredRegister` für
  den Entity-Typ).
- `config/SteveConfig` (`ForgeConfigSpec` → `ModConfigSpec`, Registrierung via
  `ModContainer#registerConfig` im Mod-Konstruktor).
- `entity/SteveEntity` (Attribute via `createAttributes`, AI-Goals-API, Spawn-Registrierung),
  `entity/SteveManager`.
- `command/SteveCommands` (Brigadier — weitgehend stabil, kleinere Signatur-Änderungen).
- `event/ServerEventHandler` (NeoForge-Events).
- Pure-Java-Pakete (`llm/`, `llm/async/`, `llm/resilience/`, `di/`, eigener `event/`-Bus)
  ziehen **unverändert** um.
- **Verify:** Mod lädt, `/steve …` funktioniert, Steve spawnt, eine Basis-Action läuft und
  erreicht Claude.

### Phase 2 — Actions & Welt-Interaktion

- 10× `action/actions/*`, `structure/*`, `memory/WorldKnowledge`, `util/ActionUtils`,
  `execution/SteveAPI` + `ActionContext`.
- Hot Spots: Blockstate/Placement, Inventar, Crafting, **Data Components** statt Item-NBT.
- **Verify:** jede Action-Kategorie funktioniert in-game.

### Phase 3 — Client-GUI / Rendering

- `client/*` (Renderer-/Model-Registrierung, `SteveOverlayScreen`, `SteveGUI`,
  `KeyBindings`, `ClientSetup`, `ClientEventHandler`).
- Hot Spots: Render-Pipeline, `GuiGraphics`, Key-Mapping-Registrierung, `RenderType` —
  am stärksten veränderter Bereich.
- **Verify:** Overlay/GUI öffnet, Steve rendert korrekt.

## Bewusst nicht geändert

- Alle LLM-Provider (Claude bleibt Default — bestehende Arbeit unangetastet), Async-Schicht,
  Resilience4j (reines Java), Gson (von MC bereitgestellt), DI, eigener Event-Bus.

## Risiken

- **GraalVM polyglot + Java 25:** Kompatibilität unbekannt → ggf. Versions-Bump oder
  Alternative nötig (Phase-0-Check).
- **Wissensgrenze (Jan 2026):** 26.1-spezifische APIs werden live recherchiert; iterative
  Compile-Fix-Zyklen pro Phase.
- **Data-Components-Migration:** überall dort, wo Item-NBT verwendet wird.
- **Client-Rendering:** der am stärksten veränderte Bereich; höchste Unsicherheit.

## Tests / Success-Kriterien

- Bestehende Pure-Java-Unit-Tests (z. B. `AsyncClaudeClientTest`) bleiben über alle Phasen
  grün: `./gradlew test`.
- MC-Laufzeitverhalten: manuelle In-game-Verifikation pro Phase (Nutzer).
- Pro Phase gilt „fertig", wenn: kompiliert + bestehende Tests grün + In-game-Verifikation
  durch den Nutzer bestätigt.
